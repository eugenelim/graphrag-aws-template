## Objective

The `graphrag.observability` module implements the OTEL observability decision from ADR-0015 for the MCP Lambda (`graphrag.mcp`) and the ingestion Fargate task. It is the Python side of the `packages/graphrag/otel-instrumentation` work item. It provides the reusable instrumentation primitives that `graphrag.mcp`, `graphrag.routing`, and the ingestion task call; it does not create most spans itself and does not own the Terraform (see scope boundary below).

The deliverable is:

1. **Content-capture filter (`ContentCaptureFilterExporter`)** — a `SpanExporter` decorator that strips, before delegating to the wrapped exporter, every attribute whose key is in **either** of two sets: the manual `DENY_SET` `{"question.text", "query.text", "sparql.query", "document.content", "chunk.text"}`, **and** `AUTO_CAPTURE_KEYS` — the OTEL semantic-convention keys ADOT's `boto3`/`urllib3` auto-instrumentation populates with request content (`db.statement`, `db.query.text`, `http.url`, `url.full`, `url.query`, `http.request.body`, and the gen-AI prompt/content attributes). It is the in-process content-capture guarantee for **every exporter this module registers** (tests, local/console, the ingestion task, and any non-ADOT context where the module owns the provider). **It is an exporter wrapper, not a `SpanProcessor`**, because OTEL Python's `SpanProcessor.on_end` receives an immutable `ReadableSpan` and cannot strip attributes (see Assumptions). On the ADOT-owned Lambda pipeline the module does **not** own the exporter (ADOT installs the global provider; see Scope boundary + Assumptions), so enforcement there is the collector-side attribute processor + config-level capture suppression (item 6). `DENY_SET` is a single canonical constant equal to the five names ADR-0015 item 6 defines and `spec-mcp-tool-server` AC5's linter checks; a pin test (AC6) holds it equal to those literals.

2. **OTEL bootstrap (`configure_observability(service_name)`)** — in non-ADOT contexts (tests, local, ingestion) it sets up the `TracerProvider` and wraps every exporter it registers in `ContentCaptureFilterExporter`; in Lambda the ADOT-installed provider is used as-is (a module `set_tracer_provider()` is a no-op) and the collector processor is the enforcement point (see item 1 + Assumptions). Offline-safe either way: callable with no AWS credentials and no reachable OTLP endpoint, raising no export error (the ADOT collector drops silently in Lambda; tests use an in-memory exporter).

3. **EMF metrics helper (`emit_tool_metrics`)** — a thin wrapper over `aws_embedded_metrics` that emits the five ADR-0015 metrics with their exact names, types, and dimensions. Falls back to plain-log output when no EMF sink is configured (offline).

4. **Structured JSON logging (`configure_json_logging`)** — configures `python-json-logger` so all log output is `{"timestamp", "level", "name", "message", "request_id"}`. Shared by the MCP Lambda and the ingestion Fargate task.

5. **Span vocabulary + leg helper (`traced_leg`)** — the canonical span-name and `SpanKind` constants for the `ask` trace (`mcp.ask` root, `routing.rule_router`, `routing.bedrock_router`, `retrieval.<strategy>`) and a context-manager helper for creating the manual OTEL leg spans. This spec **provides** the helper and the names and verifies them directly (AC4). Adopting them in the `ask` orchestrator / router is a seam owned by `spec-multi-strategy-routing`; that spec today models its own caller-visible per-leg timing (`LegSpan.latency_ms`) and does not yet reference `traced_leg` — see the timing-reconciliation seam in Assumptions.

6. **Auto-instrumentation capture suppression** — ADOT auto-instruments `boto3` (Bedrock) and `urllib3` (the Neptune SPARQL HTTPS endpoint), which by default can record the SPARQL statement (question-derived entities), request URLs, and Bedrock prompts under `AUTO_CAPTURE_KEYS`. This is the **single largest content-capture vector** and is invisible to the AC5 static linter (which reads only hand-authored source; the auto attributes are set by the ADOT layer at runtime). The module (a) configures the `boto3`/`urllib3` instrumentation to **not capture** statements, request bodies, URL query strings, or prompt content — the primary control — via the instrumentations' capture options where the module owns setup, and via `OTEL`/instrumentation env vars on the Lambda (asserted by AC7); and (b) lists `AUTO_CAPTURE_KEYS` in both the in-process filter and the collector-processor delete-set as the backstop. Suppression-at-config plus strip-at-export are complementary — neither alone is trusted.

### Scope boundary

This spec owns the Python `graphrag.observability` module and the `test_otel_conventions.py` conventions test (new — the location ADR-0015 names, in the `graphrag` package's tests tree). It does **not** own:

- **The Terraform ADOT layer wiring** — the layer ARN, `AWS_LAMBDA_EXEC_WRAPPER=/opt/otel-instrument`, and the `AWSXRayDaemonWriteAccess` grant on `mcp_lambda_role`. That is `infra-tf/mcp-otel-lambda`, constrained by ADR-0015 directly. AC7 below cross-references its `test_plan.py` assertion; it does not re-implement it.
- **The tool handlers** (`spec-mcp-tool-server`) or **the routing logic** (`spec-multi-strategy-routing`). This module provides the helpers those specs import; it does not implement their retrieval or routing.
- **The ADOT collector's attribute-processor config** — on the ADOT-owned Lambda pipeline this is the content-capture **enforcement point** (an `attributes` delete processor over the deny-set), owned by `infra-tf/mcp-otel-lambda` and verified end-to-end by AC8. In Lambda the ADOT layer installs the global `TracerProvider` and the OTLP export pipeline before the handler runs, so the module's `ContentCaptureFilterExporter` is **not** on that path; it guarantees the exporters the module itself registers (tests, local/console, ingestion) — see Assumptions.

### Content-capture layering (why three mechanisms, not one)

The content-capture policy is enforced at three points, each catching a different failure mode — this is deliberate defence-in-depth, not redundancy:

| Layer | Mechanism | Owner | Catches / limits |
|-------|-----------|-------|---------|
| Config suppression | `boto3`/`urllib3` instrumentation capture-off (options + Lambda env) | **this spec** (item 6) + `infra-tf/mcp-otel-lambda` (AC7) | stops the auto keys being *set* at all — the primary control for the `AUTO_CAPTURE_KEYS` vector |
| Author-time | Static linter over `_tools.py`/`_orchestrator.py`/`_generator.py` | `spec-mcp-tool-server` AC5 (interim) | a developer typing a `DENY_SET` key into a manual span call. **Does not** catch `AUTO_CAPTURE_KEYS` (set at runtime, not in source) or span calls in files outside those three (e.g. the router) — an author-time aid, not load-bearing |
| Runtime (in-process) | `ContentCaptureFilterExporter` (strips `DENY_SET` ∪ `AUTO_CAPTURE_KEYS`) | **this spec** (AC1) | **the load-bearing runtime control on module-owned paths** — source-agnostic, so it strips manual, router, and auto-instrumented spans alike; active on exporters this module registers (tests, local/console, ingestion) |
| Export (ADOT/Lambda) | ADOT collector `attributes` delete processor (`DENY_SET` ∪ `AUTO_CAPTURE_KEYS`) | `infra-tf/mcp-otel-lambda` (offline-asserted by AC7, end-to-end by AC8) | **the load-bearing runtime control on the ADOT-owned Lambda pipeline**, where ADOT — not this module — owns the exporter |

ADR-0015 records the static linter as "interim coverage until `spec-otel-observability` ships the programmatic filter." The linter is **not** the load-bearing control — it is a fast author-time aid with a limited three-file scope that cannot see runtime auto keys or router-file spans. The load-bearing runtime controls are the in-process filter (module-owned paths) and the collector processor (Lambda), both of which are span-source-agnostic and strip `DENY_SET` ∪ `AUTO_CAPTURE_KEYS`. An **allowlist** (deny-by-default, permit only enumerated benign keys) was considered and rejected: ADOT emits many benign, useful attributes (`db.system`, `http.status_code`, `net.peer.name`) that an allowlist would suppress, degrading the telemetry the ADR wants; a denylist of the content-bearing keys plus config-level capture-off keeps the useful signal while removing the sensitive one. `DENY_SET` and the linter's name list are held identical by each side's own pin test against the ADR-0015 item 6 literals — not by one importing the other (the linter ships first, as a static string search).

## Boundaries

### Always do

- Strip every `DENY_SET` **and `AUTO_CAPTURE_KEYS`** attribute key before any span leaves an exporter this module registers — `configure_observability()` wraps **every** exporter it registers (`InMemorySpanExporter` in tests, `ConsoleSpanExporter` locally, and any exporter the module owns) in `ContentCaptureFilterExporter`; no module-registered exporter is left un-wrapped. On the ADOT-owned Lambda pipeline the module does not register the exporter — the ADOT collector attribute processor (offline-asserted by AC7, end-to-end by AC8) is the enforcement point there.
- Configure the `boto3`/`urllib3` auto-instrumentation to **not capture** SPARQL statements, request bodies, URL query strings, or Bedrock prompt content — the `AUTO_CAPTURE_KEYS` carry question-derived content and are invisible to the AC5 linter, so config-level suppression is the primary control and the filter/collector are the backstop.
- Emit the five ADR-0015 metrics with their exact names, types, and dimensions via `aws_embedded_metrics`: `mcp.tool.duration_ms` (Histogram, `tool_name`), `mcp.tool.error_count` (Count, `tool_name`+`error_type`), `routing.decided_by.bedrock.fraction` (Gauge), `retrieval.neptune.duration_ms` (Histogram, `strategy`), `retrieval.opensearch.duration_ms` (Histogram, `strategy`).
- Format all log output as structured JSON with the keys `timestamp`, `level`, `name`, `message`, `request_id`.
- Keep `configure_observability()`, `ContentCaptureFilterExporter`, `configure_json_logging()`, and `DENY_SET` importable and callable with no AWS credentials, no network, and no reachable OTLP endpoint — the offline path raises no export error.
- Name manual spans per the ADR-0015 span model and set the correct `SpanKind`: `routing.rule_router` / `routing.bedrock_router` as `INTERNAL`, `retrieval.<strategy>` as `CLIENT`.
- Emit EMF metrics unconditionally — trace sampling never suppresses a metric.

### Ask first

- Adding a span-attribute key that could carry document-derived or user-supplied text — it must be reviewed against `DENY_SET` and added to it if it can carry content.
- Changing any metric name, type, or dimension — CloudWatch Logs Insights queries, EMF metric extraction, and any future alarm pin them.
- Removing a key from `DENY_SET` — the content-capture policy is a security control; a removal needs security review, not just a code change.
- Adding a standing-cost observability backend (a standalone OTEL collector ECS task, AWS Managed Prometheus, a third-party APM) — ADR-0015 rejected these against the cost floor (`biz-ops-budgets-threshold-above-standing-floor`).
- Adding OTEL trace spans to the ingestion Fargate task — ADR-0015 defers ingestion-path tracing; the ingestion task gets structured JSON logging only for ini-002.

### Never do

- Record question text — or any `DENY_SET` / `AUTO_CAPTURE_KEYS` content — in a span attribute, a log field at INFO level or above, or an EMF metric dimension.
- Enable statement/payload/URL-query/prompt capture in the ADOT or OTEL auto-instrumentation config — auto-instrumentation must run with content capture off (item 6).
- Populate `error_type` (or any EMF metric dimension) from an exception message or other free text — dimension values are bounded (exception class name or error code only); a free-value dimension both leaks question-derived content into CloudWatch and explodes metric cardinality (a cost / availability amplifier).
- Rely on `SpanProcessor.on_end` to strip attributes — `on_end` receives an immutable `ReadableSpan`; stripping there is a silent no-op. Use `ContentCaptureFilterExporter`.
- Register a span exporter from this module that is not wrapped in `ContentCaptureFilterExporter` — a module-registered un-wrapped exporter is a content-capture bypass. (The ADOT-owned Lambda exporter is out of the module's hands; the collector processor covers it.)
- Import boto3 or botocore in `_content_filter.py`, `_logging.py`, or the `DENY_SET`/span-vocabulary modules — only EMF/OTLP export touches AWS; the filter, the JSON-log config, and the vocabulary are AWS-free and offline-testable.
- Fail Lambda startup or the offline test suite when the X-Ray/OTLP endpoint is unreachable — export drops silently; startup proceeds. (Production export-failure *detection* is out of this module's scope — silent drop is by design for timing telemetry, which is separate from the security audit log; a broken pipeline is surfaced by a CloudWatch alarm on the ADOT collector, tracked as backlog `otel-collector-export-failure-alarm`.)
- Add a trace-sampling configuration that suppresses EMF metrics — metrics are emitted regardless of whether the request's trace is sampled.

## Assumptions

- Technical: `graphrag.observability` lives in `packages/graphrag/src/graphrag/observability/`; the conventions test is `test_otel_conventions.py` (new — the location ADR-0015 item 6 names) with additional per-file tests under `packages/graphrag/tests/observability/`.
- Technical: **OTEL Python `SpanProcessor.on_end(span)` receives an immutable `ReadableSpan`** with no attribute mutation API. The content-capture filter is therefore realized as a `SpanExporter` decorator that filters attributes in `export()`, not as an `on_end` mutation. This is a deliberate refinement of ADR-0015 item 6's "SpanProcessor" wording; the ADR's own Confirmation test ("export the span to an in-process exporter, assert the attribute is absent from the exported span data") describes exporter-observed behaviour, which the decorator satisfies exactly. **Verify against the pinned `opentelemetry-sdk` version at implementation** — if a future SDK exposes mutable-span processors, the exporter decorator remains correct and is the more defensive choice regardless.
- Technical: The new dependencies — `opentelemetry-sdk`, `opentelemetry-exporter-otlp-proto-grpc`, `aws-embedded-metrics`, `python-json-logger` — go in a new `[observability]` group under `[project.optional-dependencies]` in the root `pyproject.toml` (mirroring the existing `infra` / `ingest` / `dev` groups). The `AWSOpenTelemetryDistro` auto-instrumentation and its bundled collector are supplied by the **ADOT Lambda layer** at runtime (owned by `infra-tf/mcp-otel-lambda`), not by a pip dependency; offline tests need only `opentelemetry-sdk`.
- Technical: New `[[tool.mypy.overrides]]` entries with `ignore_missing_imports = true` are required for `opentelemetry.*`, `aws_embedded_metrics.*`, and `pythonjsonlogger.*` (the same pattern the existing config uses for `rdflib` / `pyshacl`).
- Technical: **`AUTO_CAPTURE_KEYS` is a second canonical constant** alongside `DENY_SET`, listing the OTEL semantic-convention keys ADOT's `boto3`/`urllib3` auto-instrumentation can populate with request content: `db.statement`, `db.query.text`, `http.url`, `url.full`, `url.query`, `http.request.body`, and the gen-AI prompt/content attributes (e.g. `gen_ai.prompt`). The exact key list is confirmed against the pinned `opentelemetry-instrumentation-*` versions at implementation (the semantic conventions are still evolving — hence config-level capture-off is the primary control and the key-strip is the backstop, not the reverse). The capture-off mechanism (`urllib3`/`botocore` instrumentation options where the module owns setup; `OTEL_PYTHON_*` / instrumentation env vars on the Lambda) is likewise version-confirmed at implementation.
- Technical: **`AWSXRayDaemonWriteAccess` least-privilege note.** X-Ray trace-ingest actions (`xray:PutTraceSegments`, `xray:PutTelemetryRecords`, `xray:GetSamplingRules`, `xray:GetSamplingTargets`) do not support resource-level scoping, so the AWS-managed policy is effectively equivalent to a scoped inline policy of the same four actions; the managed policy is accepted on that basis rather than silently. If a future review prefers an explicit inline policy for auditability, the action set is those four. (Mirrors the on-record IAM-scoping discipline of backlog `neptune-sparql-dropgraph-iam-action-verify`.)
- Technical: `aws_embedded_metrics` flushes EMF JSON to stdout via its environment-detected sink (`LambdaEnvironment` in Lambda, `LocalEnvironment` elsewhere). The AC2 test captures the serialized EMF payload via the library's local sink; the exact capture call is an implementation detail — the contract is the metric **name, unit, and dimensions**, not the capture mechanism.
- Technical: `configure_observability()` is called once at MCP Lambda module import (in `graphrag.mcp`'s SDK-initialization point, per `spec-mcp-tool-server` Assumptions) and once at ingestion-task startup for logging only (no tracer). Auto-instrumentation of `boto3`/`urllib3` is activated by the ADOT layer's `AWS_LAMBDA_EXEC_WRAPPER`, not by this module — this module owns the manual leg spans, the content filter, the metrics, and the log format.
- Technical: **ADOT owns the global `TracerProvider` in Lambda.** The layer's `AWS_LAMBDA_EXEC_WRAPPER=/opt/otel-instrument` auto-configures the global provider and the OTLP→collector export pipeline before the handler module runs; a later `set_tracer_provider()` from this module is a no-op (OTEL forbids overriding an installed provider). Therefore on the Lambda path the module's `ContentCaptureFilterExporter` is **not** on the active export pipeline — production content-capture enforcement there is the ADOT collector's `attributes` delete processor (owned by the `infra-tf/mcp-otel-lambda` work item, verified end-to-end by AC8). The module's filter is authoritative for exporters it registers (tests, local/console, ingestion) and for any non-ADOT context where the module owns the provider. The `traced_leg` manual spans created under the ADOT provider are covered by the collector processor + capture-off config (offline-asserted by AC7, end-to-end by AC8); the AC5 linter is an author-time aid, not the load-bearing control (see the layering table).
- Product: The interim static-linter coverage (`spec-mcp-tool-server` AC5) and this spec's in-process filter **coexist** — shipping the filter does not remove the linter. Because the linter ships first (before this module exists) and is a static string search over source text, it cannot import `DENY_SET`; instead the linter's name list and `DENY_SET` are each pinned to the five ADR-0015 item 6 canonical names by their own tests. Consolidating them onto a shared import once both have shipped is a possible follow-up, not a claim of this spec.
- Product: This spec delivers `traced_leg` + the span-name/kind constants and verifies them directly (AC4), so the contract holds without the router present. **Timing-reconciliation seam:** `spec-multi-strategy-routing` today records per-leg timing in its own caller-visible `StrategyTrace` (`LegSpan.latency_ms`, `router_latency_ms`) and does not reference `traced_leg`; ADR-0015 §Context states the strategy trace "carries no timing" and that OTEL adds timing on top. Whether the OTEL `retrieval.<strategy>` span's wall-clock duration and `LegSpan.latency_ms` are one measurement or two parallel ones is a reconciliation owned by `spec-multi-strategy-routing` — this spec neither asserts that sibling consumes `traced_leg` today nor edits it; it flags the seam so the router spec closes it when built.
- Product: Sampling is X-Ray default reservoir (10 req/s + 5% tail) per ADR-0015 — no sampling configuration is authored in this module; EMF metrics are unaffected by sampling.
