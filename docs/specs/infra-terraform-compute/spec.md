- **Shape:** data (compute infrastructure; no application logic)

> **Spec contract:** this document defines "done" for the compute tier of the Terraform
> migration. Load-bearing invariants: Function URL auth is `AWS_IAM` (never `NONE`); the
> invoke-URL permission names the `var.invoker_role_arn` principal (never `*`); the
> SmokeProbe Lambda's Neptune grant is full R/W (the query Lambda's is read-only per
> ADR-0004, already scoped in data+IAM); ECR repository has `force_delete = true` so
> `terraform destroy` leaves no billable resources.

> **Compute tier** ‚Äî ECS/Fargate ingestion task, 3 Lambda functions (smoke, vector-smoke,
> query), Function URL + permission, Budget alarm, ECR repository, 4 CloudWatch log
> groups ‚Äî translated from `apps/infra/stacks/graphrag_stack.py` to
> `apps/infra-tf/{compute,lambda,budget}.tf`. A new IAM role for the smoke Lambda
> execution (`smoke_probe_role`) is also created here (in CDK it was auto-generated by
> the Lambda construct when `add_to_role_policy()` was called).

## Objective

Provision the compute layer so the full system can be applied, probed, and destroyed
in a single `terraform apply` / `terraform destroy` cycle. The deliverables are
`compute.tf`, `lambda.tf`, and `budget.tf`. Once this spec's ACs pass, the combined
live acceptance test (`terraform apply` ‚Üí smoke probe invocations ‚Üí `terraform destroy`)
proves the migration end-to-end.

Load-bearing engineering points: (1) Function URL auth type is `AWS_IAM` ‚Äî the CDK
test `test_function_url_is_iam_auth` has a Terraform-equivalent plan assertion in
`infra-terraform-verification`; (2) invoke-URL permission names `var.invoker_role_arn`
as principal, never `*`; (3) SmokeProbe Lambda gets full Neptune R/W (same as
IngestionTask ‚Äî it runs live insert+retrieve, not just reads); (4) all log groups
have `retention_in_days = 7` and are stack-managed so `terraform destroy` removes them
without manual cleanup. NOTE (contract-acquisition, provider 5.x): `aws_cloudwatch_log_group`
has **no** `force_destroy` argument ‚Äî `terraform destroy` deletes a managed log group and
its events by default, so the CDK `RemovalPolicy.DESTROY` equivalent is simply managing
the group as a resource with no `prevent_destroy`/`skip_destroy` (not a `force_destroy`
flag, which the provider rejects).

## Boundaries

### Always do

- **ECS cluster + ECR repository in the VPC.** `aws_ecs_cluster` in the VPC.
  `aws_ecr_repository` with `force_delete = true` (matches CDK `empty_on_delete=True`).
  The ECR repo is the ingestion container image source; it must be destroyable by
  `terraform destroy` without a manual `ecr delete-repository --force` step.
- **Fargate task definition: cpu=512, memory=1024, network_mode=awsvpc,
  requires_compatibilities=["FARGATE"].** Task role = `ingestion_task_role` (created in
  data+IAM tier). Execution role = a new `ecs_task_execution_role` with the
  `AmazonECSTaskExecutionRolePolicy` managed policy (for ECR pull + CloudWatch Logs).
  Container env vars exactly match the CDK container: `NEPTUNE_ENDPOINT`,
  `OPENSEARCH_ENDPOINT`, `CORPUS_BUCKET`, `SCHEMA_EXTRACTION=false`.
- **4 CloudWatch log groups with `retention_in_days = 7`, stack-managed & destroyable.**
  CDK uses `RemovalPolicy.DESTROY` + `RetentionDays.ONE_WEEK` for all 4 log groups
  (IngestionLogs, SmokeProbeLogs, VectorSmokeProbeLogs, QueryLambdaLogs). In Terraform
  the equivalent is managing each group as an `aws_cloudwatch_log_group` resource with
  `retention_in_days = 7` and **no** `prevent_destroy`/`skip_destroy`: `terraform destroy`
  deletes the group (and its log events) by default. `aws_cloudwatch_log_group` has no
  `force_destroy` argument in provider 5.x (confirmed via contract-acquisition) ‚Äî none
  is needed and setting it fails `terraform validate`.
- **SmokeProbe Lambda: Python 3.12, timeout=60s, VPC-attached, handler
  `graphrag.smoke_lambda.lambda_handler`, environment `NEPTUNE_ENDPOINT`.** Uses a
  new `smoke_probe_role` (ECS auto-generates one in CDK; in Terraform it must be
  explicit). `smoke_probe_role` has: trust `lambda.amazonaws.com`, managed policy
  `AWSLambdaVPCAccessExecutionRole`, inline policy `neptune-full-rw` (connect +
  ReadDataViaQuery + WriteDataViaQuery + DeleteDataViaQuery ‚Äî the smoke probe runs
  insert+retrieve, not just reads).
- **VectorSmokeProbe Lambda: Python 3.12, timeout=120s, VPC-attached, handler
  `graphrag.vector_smoke_lambda.lambda_handler`, role=`vector_probe_role`.**
  `vector_probe_role` created in data+IAM tier (OpenSearch + Bedrock Titan grants).
  Environment: `OPENSEARCH_ENDPOINT`.
- **QueryLambda: Python 3.12, timeout=120s, memory_size=512, VPC-attached, handler
  `graphrag.query_lambda.lambda_handler`, role=`query_role`.** Environment:
  `NEPTUNE_ENDPOINT`, `OPENSEARCH_ENDPOINT`, `SYNTHESIS_MODEL_ID`.
- **Function URL: `authorization_type = "AWS_IAM"`.** On the QueryLambda. Never `NONE`.
  Exactly one `aws_lambda_function_url` in the plan.
- **Lambda permission scoped to `var.invoker_role_arn`.** `aws_lambda_permission` with
  `action = "lambda:InvokeFunctionUrl"`, `principal = var.invoker_role_arn`,
  `function_url_auth_type = "AWS_IAM"`. Never `principal = "*"`.
- **Budget alarm: $150/mo COST, 80% ACTUAL threshold, email subscriber
  `var.budget_alarm_email`.** `aws_budgets_budget` with
  `budget_type = "COST"`, `time_unit = "MONTHLY"`, `limit_amount = "150"`,
  `limit_unit = "USD"`. Notification: `threshold = 80`, `threshold_type = "PERCENTAGE"`,
  `notification_type = "ACTUAL"`, `comparison_operator = "GREATER_THAN"`.
  Subscriber: `subscriber_email_addresses = [var.budget_alarm_email]` on the notification
  block. NOTE (contract-acquisition, provider 5.x): `aws_budgets_budget` expresses email
  subscribers as a `subscriber_email_addresses` set **inside** the `notification` block ‚Äî
  not a nested `subscriber { subscription_type = "EMAIL", address = ... }` block (that is
  the CloudFormation/CDK shape). Semantic intent (an email subscriber = `var.budget_alarm_email`)
  is unchanged.
- **All Lambda functions are VPC-attached (private isolated subnets).** No Lambda may
  have a public endpoint other than the IAM-auth Function URL on QueryLambda.
- **All Lambda log groups are stack-managed** (`logging_config.log_group` on
  `aws_lambda_function`), pointing at the pre-created `aws_cloudwatch_log_group`
  resources ‚Äî not the Lambda-auto-created `/aws/lambda/<fn>` group. This ensures
  `terraform destroy` removes the log group.
- **`teardown-first`: no `prevent_destroy` on any resource.** ECR repo, log groups,
  ECS cluster, and budget must all be destroyable by `terraform destroy`.

### Ask first

- Changing the Lambda runtime (Python 3.12 is pinned to match the CDK stack).
- Changing Lambda timeout values (60s smoke, 120s vector-smoke/query are pinned).
- Changing the Fargate CPU/memory (512/1024 is pinned to match CDK).
- Changing the budget limit or threshold.

### Never do

- **Never `authorization_type = "NONE"` on the Function URL** ‚Äî the CDK test
  `test_function_url_is_iam_auth` enforces AWS_IAM; the Terraform plan assertion
  must pass the same check.
- **Never `principal = "*"` on the Lambda permission** ‚Äî CDK test
  `test_function_url_invoke_permission_scoped_to_named_principal` enforces a named
  principal (the InvokerRoleArn).
- **Never attach Neptune write actions to `query_role`** ‚Äî ADR-0004 hard rule;
  the query Lambda's Neptune grant is read-only and is defined in the data+IAM tier.
  The compute spec must not add write actions to it.
- **Never hardcode account IDs or region strings** ‚Äî use `data.aws_caller_identity`
  and `var.aws_region`.

## Testing Strategy

- **AC1‚ÄìAC9 + AC11 ‚Äî goal-based check.** Verified by `terraform validate` + `terraform
  fmt -check` + `terraform plan`/`terraform show -json` output. The full assertion suite
  is in `infra-terraform-verification`; this spec's gate is `validate` + `fmt -check` +
  targeted plan-JSON checks for the load-bearing security invariants (Function URL auth,
  Lambda permission principal, exactly-one Function URL, SmokeProbe 4 Neptune actions,
  query_role no Write/Delete) plus the AC11 negative check (a wildcard/`:root`
  `invoker_role_arn` must fail `terraform validate`).
- **AC10 ‚Äî infra/deploy (live). Deferred to `infra-terraform-verification`.** The live
  cycle below runs once, as part of the verification tier's single full-stack live run
  (that spec owns the readiness-wait `probe.sh` and re-applies network+data+compute
  together ‚Äî a second apply/destroy here would be redundant). Backlog anchor:
  `terraform-compute-live-cycle`. The sequence:
  - `terraform apply` completes without error (Neptune/OpenSearch reach AVAILABLE).
  - Invoke SmokeProbe: `aws lambda invoke --function-name <SmokeProbeName> /tmp/out.json`
    and confirm exit code 0 + `"success"` in `/tmp/out.json`.
  - Invoke VectorSmokeProbe: same pattern.
  - Invoke QueryLambda via Function URL: `curl -s --aws-sigv4 "aws:amz:..."
    <QueryFunctionUrl>` confirms JSON response, no 4xx/5xx.
  - `terraform destroy` completes without error; confirm no ECR repo, log groups, or
    Neptune cluster remain.

Gates: `terraform fmt -check`, `terraform validate`, plan JSON Function URL auth check,
plan JSON Lambda permission principal check.

> **IaC scanner gap (accepted).** No policy-as-code / CSPM scanner (Checkov / tfsec /
> KICS) is wired against `apps/infra-tf/` yet ‚Äî the config-misconfig class for this tier
> (IAM grants, Function URL, ECR, SG wiring) is judged by review only, and the spec-stage
> `security-reviewer` pass reasoned the diff by hand (`degraded: no scanner`). Wiring a
> Terraform scanner in CI is tracked as `infra-terraform-scanner-ci` (workspace.toml
> backlog). Related latent items the scanner would own: the unmanaged VPC default SG
> (allow-all intra-group ‚Äî not on our ENI path, which uses the explicit per-compute SGs)
> and ECR tag-immutability (deliberately mutable `:latest`, see AC1 note).

## Acceptance Criteria

- [x] **AC1 ‚Äî ECS cluster + ECR repository with `force_delete = true`.** *(goal-based check)*
  `aws_ecs_cluster` present in plan. `aws_ecr_repository` with `force_delete = true`.

  `image_tag_mutability = "IMMUTABLE"` is intentionally NOT set here (it would break the

  `aws_iam_role.ecs_task_execution_role` (trust: `ecs-tasks.amazonaws.com`) present in plan.

  resource ARN (reuses `local.neptune_rw_policy` from iam.tf ‚Äî one source of truth, not a
  re-inlined copy). The full R/W set is CDK parity: the probe **inserts** a test node,
  **retrieves** it, then **deletes** it (cleanup) ‚Äî hence `WriteDataViaQuery` +
  `DeleteDataViaQuery`, not reads alone. The `query_role` in the plan must NOT have
  Write/Delete Neptune actions (verified by plan-assertion test ‚Äî the ADR-0004 invariant
  is unchanged by this spec).

- [x] **AC7 ‚Äî Function URL: `AWS_IAM`, exactly one; Lambda permission: named principal.** *(goal-based check)*
  `aws_lambda_function_url.query_url` with `authorization_type = "AWS_IAM"`. Exactly one
  `aws_lambda_function_url` in the plan. `aws_lambda_permission.query_url_invoke` with
  `action = "lambda:InvokeFunctionUrl"`, `principal = var.invoker_role_arn`,
  `function_url_auth_type = "AWS_IAM"`. No `aws_lambda_permission` with `principal = "*"`.

- [x] **AC8 ‚Äî Budget alarm: $150/mo, 80% ACTUAL threshold, email subscriber.** *(goal-based check)*
  `aws_budgets_budget` with `limit_amount = "150"`, `limit_unit = "USD"`,
  `budget_type = "COST"`, `time_unit = "MONTHLY"`. Notification block:
  `threshold = 80`, `threshold_type = "PERCENTAGE"`, `notification_type = "ACTUAL"`,
  `comparison_operator = "GREATER_THAN"`. Subscriber:
  `subscriber_email_addresses = [var.budget_alarm_email]` on the notification block
  (provider 5.x shape ‚Äî not a nested `subscriber` block; see the "Always do" note).

- [x] **AC9 ‚Äî `terraform plan` for the compute tier completes without error; all 12 outputs
  populated.** *(goal-based check)* With the data+IAM tier applied (or mocked),
  `terraform plan` exits 0 and shows all 12 output values populated (no `null` stubs).

- [x] **AC10 ‚Äî Live: `terraform apply` + smoke probes + clean `terraform destroy`.** *(infra/deploy ‚Äî live)* (deferred: terraform-compute-live-cycle)
  `terraform apply` succeeds; SmokeProbe/VectorSmoke/Query invocations pass; `terraform
  destroy` succeeds cleanly. See Testing Strategy ¬ß AC10. Deferred to the
  `infra-terraform-verification` tier, which
  owns the readiness-wait `probe.sh` and runs the single full-stack live cycle
  (apply ‚Üí smoke ‚Üí destroy); running it here would duplicate that expensive cycle.

- [x] **AC11 ‚Äî `invoker_role_arn` validation rejects root, wildcard, and non-role principals.** *(goal-based check)*
  The `var.invoker_role_arn` validation in `variables.tf` is end-anchored and admits only
  a role ARN with a role-name/path body ‚Äî regex
  `^arn:aws:iam::[0-9]{12}:role/[A-Za-z0-9+=,.@_/-]+$`. A wildcard body
  (`arn:aws:iam::123456789012:role/*`), a root ARN (`:root`), or a non-role principal
  fails `terraform plan`/`validate`. This is the fail-fast guard on the public-inbound
  Function URL principal (defence-in-depth alongside the Lambda `AddPermission` API's own
  wildcard rejection). Verified by the T7 negative check (a wildcard-body / `:root` ARN
  must exit non-zero).

## Assumptions

- Technical: Lambda code packaging ‚Äî each Lambda function references a pre-built zip
  at `apps/graphrag/dist/graphrag.zip` (produced by the `apps/graphrag` build step).
  The Terraform `aws_lambda_function` uses `filename` for local file or S3-based
  deployment. The zip must exist before `terraform apply`; the spec does not define the
  build step (out of scope; the CI/CD pipeline owns it).
- Technical: `aws_lambda_function` `logging_config` block (pointing at a named log
  group) is available in AWS provider >= 5.40; the `versions.tf` `~> 5.0` constraint
  satisfies this (source: contract-acquisition during implementation).
- Technical: `aws_budgets_budget` `notification` uses `threshold` as a number (not
  string) in the provider >= 5.0; verify argument types against schema during
  implementation.
- Technical: the data+IAM tier outputs (`aws_iam_role.ingestion_task_role.arn`,
  `aws_iam_role.vector_probe_role.arn`, `aws_iam_role.query_role.arn`,
  `aws_neptune_cluster.main.cluster_resource_id`) are available in the Terraform state
  (workspace.toml dependency chain ensures data+IAM is applied or at least planned
  before compute builds).
- Process: the combined live acceptance test (`apply` ‚Üí smoke ‚Üí `destroy`) is the
  first live deploy of the full Terraform stack; it supersedes the CDK live deploy ACs
  for this feature branch.
